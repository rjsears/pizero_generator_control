# Cloudflare Tunnel Real IP Fix

## Diagnostic Checklist (Do This First)

When asked to check/fix Cloudflare Tunnel real IP issues, follow these steps:

### Step 1: Confirm the Problem Exists

```bash
# Check nginx logs - look for Docker IPs (172.x.x.x) instead of real client IPs
docker logs <nginx_container> --tail 20

# If you see entries like:
# 172.18.0.1 - - [10/Feb/2026:16:00:00 +0000] "GET / HTTP/1.1" ...
# ^^^^^^^^^^^ This is a Docker IP - THE PROBLEM EXISTS

# If you see real external IPs (like 98.173.x.x), the fix is already in place
```

### Step 2: Check if real_ip Config Exists in nginx

```bash
# Check if nginx has the real_ip configuration
docker exec <nginx_container> grep -A5 "real_ip" /etc/nginx/nginx.conf

# If this returns nothing or no "set_real_ip_from" lines, THE FIX IS NEEDED
```

### Step 3: Find Where nginx.conf Comes From

```bash
# Check if nginx.conf is mounted from host
docker inspect <nginx_container> --format '{{json .Mounts}}' | jq .

# Look for a bind mount like:
# "Source": "/path/to/nginx.conf" -> this is where the file lives on host

# IMPORTANT: Check if there's a setup.sh or similar script that GENERATES nginx.conf
# If so, you must add the fix to the SCRIPT, not just the file
grep -r "nginx.conf" *.sh  # or similar to find generation scripts
```

### Step 4: Apply the Fix

See "The Fix" section below. Remember:
- If nginx.conf is a static file: edit the file directly
- If nginx.conf is generated by a script: edit the SCRIPT's heredoc/template

### Step 5: Verify the Fix Works

```bash
# Reload nginx
docker exec <nginx_container> nginx -t && docker exec <nginx_container> nginx -s reload

# Make a request through Cloudflare and check logs
docker logs <nginx_container> --tail 5

# You should now see real client IPs, not 172.x.x.x
```

---

## The Problem

When using Cloudflare Tunnel (cloudflared) with nginx running in Docker, nginx sees all external traffic as coming from a Docker network IP (e.g., 172.18.0.1) instead of the real client IP.

**Why this happens:**
1. External client connects to Cloudflare
2. Cloudflare routes through the tunnel to the cloudflared container
3. cloudflared container forwards to nginx container
4. Both containers are on a Docker network
5. nginx sees the cloudflared container's Docker IP, not the real client

**The consequence:**
- IP-based access control fails (all traffic appears "internal")
- Rate limiting doesn't work properly (all traffic shares one IP)
- Logs are useless for identifying clients

## The Fix

### Step 1: Add real_ip Configuration to nginx.conf

Add this block in the `http` section, BEFORE the `geo` block and `log_format`:

```nginx
http {
    include /etc/nginx/mime.types;
    default_type application/octet-stream;

    # ==========================================================================
    # Real IP Configuration for Cloudflare Tunnel
    # ==========================================================================
    # Trust Docker/internal networks as proxies (cloudflared connects from here)
    # This allows nginx to extract the real client IP from X-Forwarded-For header
    set_real_ip_from 172.16.0.0/12;
    set_real_ip_from 10.0.0.0/8;
    set_real_ip_from 192.168.0.0/16;
    set_real_ip_from 100.64.0.0/10;
    set_real_ip_from 127.0.0.1;

    # Cloudflare sets X-Forwarded-For with the real client IP
    real_ip_header X-Forwarded-For;

    # Enable recursive lookup (use rightmost untrusted IP in chain)
    real_ip_recursive on;

    # ... rest of config (log_format, geo block, etc.)
```

### Step 2: If nginx.conf is Generated by a Script

If your nginx.conf is generated by a setup script (like setup.sh), you must add the real_ip configuration to the script's heredoc/template, NOT just the file. Otherwise, your changes will be overwritten when the script runs again.

Look for something like:
```bash
cat > nginx.conf << 'EOF'
http {
    include /etc/nginx/mime.types;
    default_type application/octet-stream;
    # ADD THE real_ip CONFIG HERE
```

### Step 3: Reload nginx

```bash
# Test config first
docker exec <nginx_container> nginx -t

# If OK, reload
docker exec <nginx_container> nginx -s reload
# OR
docker restart <nginx_container>
```

## How to Verify It's Working

### Test 1: Check nginx logs

Add `$http_x_forwarded_for` to your log format temporarily:

```nginx
log_format main '$remote_addr - XFF:$http_x_forwarded_for - [$time_local] ...';
```

Reload nginx and make a request through Cloudflare:
- `XFF:-` = Cloudflare is not sending the header (problem with Cloudflare config)
- `XFF:1.2.3.4` = Header is present, real_ip should extract it

### Test 2: Check $remote_addr

After the real_ip config is applied, `$remote_addr` in logs should show the real client IP, not 172.x.x.x.

Make a request from a known external IP and verify it shows in logs.

### Test 3: Verify access control works

1. Access through Cloudflare from an IP NOT in your allowed list
2. You should get 403 Forbidden
3. Check logs - you should see the real external IP with "external" access level

## What Each Directive Does

| Directive | Purpose |
|-----------|---------|
| `set_real_ip_from` | Trust these IPs as reverse proxies. When request comes from these IPs, look at headers for real IP. |
| `real_ip_header` | Which header contains the real client IP. Cloudflare uses `X-Forwarded-For`. |
| `real_ip_recursive on` | If there are multiple IPs in the header chain, find the rightmost one that's not trusted. |

## Why X-Forwarded-For (not CF-Connecting-IP)

Cloudflare sets both headers:
- `CF-Connecting-IP`: Cloudflare-specific header with client IP
- `X-Forwarded-For`: Standard proxy header with client IP

Using `X-Forwarded-For` works for Cloudflare and is also compatible with other proxies (like Tailscale Serve if you use it).

## Common Issues

### Issue: Still seeing Docker IPs after config change
- **Cause**: nginx wasn't reloaded
- **Fix**: `docker exec <container> nginx -s reload`

### Issue: Config change doesn't persist
- **Cause**: nginx.conf is generated by a script that overwrites it
- **Fix**: Add the config to the generation script, not just the file

### Issue: All traffic blocked after change
- **Cause**: real_ip is working but geo block doesn't include the real IPs
- **Fix**: Add your allowed external IPs to the geo block

### Issue: XFF header is empty
- **Cause**: Cloudflare Tunnel not configured to forward headers
- **Fix**: Check cloudflared/tunnel configuration in Cloudflare dashboard
